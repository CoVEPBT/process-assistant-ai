Four parallel sub-agents reviewed the codebase at end of day. This is the
consolidated findings list, ranked by severity. Issues marked **P1** are
real bugs likely to bite a pilot user; **P2** are inconsistencies / smells;
**P3** are documentation / dead-code cleanups.

## P1 — Real bugs to fix soon

### 1. AU-spelling flags fall through to the process-wide banner
- **Where:** `runRulesEngine` produces flag IDs like `au-spelling-organize-objective` (~line 8767). `resolveJumpTarget` has no pattern that matches `au-spelling-*`.
- **Impact:** Every AU-spelling flag — there can be dozens per process — silently lands in the top "process-wide" banner instead of anchoring to the misspelled word inline. The user can't click to navigate.
- **Fix sketch:** Add an `au-spelling-*` regex test in `resolveJumpTarget` that pulls the location suffix from the ID and resolves it. Even falling through to the existing location-based matcher would help.

### 2. Email-address restoration breaks after multi-role splits
- **Where:** `startAutoApply` post-process block ~line 13992. The email-restoration pass uses simple index-based matching (`origActs[ai]`) with no fallback to number-matching.
- **Impact:** When the AI legitimately splits a multi-role activity (source activity 2 becomes new activities 2 and 3), the loop pulls emails from `origActs[2]` when processing new activity at index 2, fetching the wrong source activity entirely. The attachment-restoration pass at ~line 13830 *does* use number-matching with index fallback — email restoration just doesn't.
- **Fix sketch:** Mirror the pattern at line 13830 (number-match first, index fallback) in the email pass.

### 3. Activity-number restoration silently truncates after splits
- **Where:** ~line 14064: `lim = Math.min(newCount, origCount)`.
- **Impact:** If source activity 3 splits into new activities 3 and 4, the loop only iterates `min(newCount, origCount)` — so new activity 4 never gets its number restored, and subsequent activities stay renumbered. The safety net I just added will warn about activity-count growth but won't flag the numbering drift.
- **Fix sketch:** After the cap-at-min restoration, scan for unrestored activities beyond `origCount` and assign sequential numbers continuing from the last restored number.

### 4. extractText collapses multi-line NOTE bodies into a single line
- **Where:** Line ~14926: `(clone.innerText || clone.textContent || '').replace(/\s+/g, ' ').trim()`.
- **Impact:** Saving a NOTE with a multi-paragraph body via the doc preview flattens it into one line.
- **Fix sketch:** For NOTE bodies specifically, preserve line breaks. Either collapse only horizontal whitespace, or read body parts separately and rejoin with `\n`.

### 5. Title-inferred flag has no ID_TO_TARGET entry
- **Where:** `flagGroup` includes `title-inferred` (~line 9739) but ID_TO_TARGET (lines 9934-9946) doesn't.
- **Impact:** Resolves to `null` initially, only accidentally routes via the location-substring fallback at ~line 9997. Fragile.
- **Fix sketch:** Add `'title-inferred': 'doc-title'` to ID_TO_TARGET.

## P2 — Prompt contradictions and weak rules

### 6. structureNotesWithAI vs auto-apply: NOTE addition rules conflict
- **structureNotesWithAI** (line 6005): *"NEVER add NOTEs, decisions, branches, explanations, rationale, or commentary."*
- **Auto-apply PROMOTE DROPPED CONTEXT block:** explicitly *mandates* creating new NOTEs when shortening tasks past the 18-word cap.
- These run on the same document at different stages. The contradiction creates an escape hatch that the AI has been exploiting — the prompt-only invented-NOTE regression we fixed three times today suggests the model is finding ways to interpret "preserve dropped context" as license to invent.
- **Fix sketch:** Clarify in structureNotesWithAI that the no-NOTE-add rule applies only to the structuring phase. Add a hard cap on auto-apply: "Only promote ONE NOTE per dropped detail, and only when the detail came from existing source content. NEVER add a NOTE whose body is content not present in the source."

### 7. Activity heading length split-exception in tension with activity-count invariant
- **ACTIVITY HEADING LENGTH step 6** allows splitting an over-length activity into two — under three conditions (a/b/c).
- **ACTIVITY COUNT INVARIANT** says activity count must match exactly.
- These can fight each other. The prompt currently resolves it by listing multi-role-split as the "only sanctioned reason to add an activity" — but the heading-length step 6 is a second sanctioned exception that isn't explicitly listed in the activity-count rule.
- **Fix sketch:** Update the ACTIVITY COUNT INVARIANT phrasing to enumerate BOTH sanctioned exceptions (multi-role split + headinglength split with all 3 signals).

### 8. PROMOTE DROPPED CONTEXT has no per-activity cap on NOTE creation
- **Where:** ~line 13700 PROMOTE DROPPED CONTEXT rule.
- **Risk:** A tightened 18-word task could spawn 5 new NOTEs, all "promoted" content. Nothing in the prompt caps how many.
- **Fix sketch:** Add a cap: "If you would create more than 2 new NOTEs under any activity, consolidate them into 1 NOTE that answers a broader question."

### 9. Severity isn't passed through to the auto-apply prompt
- **Where:** The rules engine emits flags with severity error/warning/info. The auto-apply prompt treats all flagged issues equivalently — there's no guidance to deprioritise info-level suggestions.
- **Risk:** An info-level "task is 38 words" flag triggers the same level of rewrite as an error-level "missing process title".
- **Fix sketch:** Pass severity in the issues context block, then guide the prompt: "errors must be fixed, warnings should be addressed where the rule applies cleanly, info-level suggestions can be skipped if the fix would risk losing source content."

## P2 — Parser inconsistencies

### 10. Three parsers, three whitespace conventions
- `flattenDocxHtml` ~line 6568: collapses runs of 3+ newlines to 2.
- `appendContinuation` ~line 7182: joins with single space, no collapse check.
- `extractText` ~line 15000: collapses all runs to single space.
- **Fix sketch:** Extract a shared `normaliseWhitespace(text, collapseRuns=true)` helper used by all three.

### 11. NOTE body extraction differs across parsers
- `parseNintexPdf` ~line 7185: splits on newline first, then `?`, then sentence-ending regex.
- `parseProcedureTextExport` ~line 7512: accumulates body by appending raw-trimmed lines.
- `parseStructuredText` ~line 7787: splits on `?` only.
- **Fix sketch:** Pick one canonical NOTE split strategy and use it everywhere.

### 12. Unicode dash normalisation incomplete in extractModelFromDocPreview
- **Where:** ~line 15033. Tests `txt === '-'` and `txt === '–'` (en-dash) but not `—` (em-dash).
- **Fix sketch:** Single regex `/^[\-–—]$/.test(txt)` or test all three.

### 13. looksLikeNintexExport too eager on title-page-only PDFs
- A PDF with "Process Manager" text but no actual activities will still be routed to `parseNintexPdf`, producing an empty activities list.
- **Fix sketch:** Verify at least one `^\d+\.\d+\s+\w+` line exists in the text before treating as a procedure export.

## P2 — Race conditions and state sync

### 14. No lock on global state during auto-apply async fetch
- Between the fetch at ~line 13757 and the restoration passes at ~line 13819, the user can edit the doc preview, click Reset, or trigger another analyse. `currentModel`, `currentMermaid`, `currentBpmnXml` could be wiped while auto-apply is mid-restoration.
- **Fix sketch:** Set `window._autoApplyInProgress = true` at start, gate Reset/Analyse on that flag, clear at end.

### 15. originalSnapshot is anchored on first run only
- Intentional per tracker #52, but for the post-process safety net I added today: a second auto-apply run after manual edits will compare against the post-first-apply baseline, not the original parse. The "Heads up: 2 tasks missing" warning could fire on tasks the user themselves deleted between runs.
- **Fix sketch:** Either snapshot before EACH auto-apply (changes the Restore-original semantics), or note in the safety-net diff "vs pre-run state, not original".

### 16. Multi-role-split-aware dropped-context detection
- The dropped-context warnings (~line 13884) compare wrong activities to wrong activities after a multi-role split.
- **Fix sketch:** Use number-matching (with index fallback) the same way the attachment-restoration pass does.
