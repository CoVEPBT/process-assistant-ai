Protected system / tool / app names list for #33 follow-up

Natalie pasted the Nintex Process Manager systems/tools/apps export
(287 entries). Combined with the existing hardcoded list in the

New file: public/protected-system-names.txt
  - Single source of truth, one name per line, "#" comments allowed.
  - Sorted alphabetically (case-insensitive) for easy scanning + edit.
  - Lives in public/ so it's served as a static asset (the client
    fetches it once on first auto-apply and caches in memory).
  - Easy to edit / extend — add a new system, save the file, push.

Two consumers of the same list:

1. Auto-apply prompt — filter to source.
   On every auto-apply, the client scans the source documentText for
   names that actually appear in it. ONLY those names go into the
   prompt as a "PROTECTED SYSTEM / TOOL / APP NAMES" block:
     "The source process mentions the following named systems, tools,
      apps, documents, forms, or acronyms from the RMIT / CoVE / Nintex
      Process Manager registry. They are CANONICAL nouns — the reader
      knows them by these exact names. Keep them EXACTLY as written in
      the source. Do not abbreviate them, do not expand them, do not
      generalise them ('SAMS' must not become 'the system'; 'A&T
      schedule' must not become 'the schedule'), and do not drop them
      when tightening a sentence.
        - SAMS
        - A&T schedule
        - ..."
   Filter-by-source means the prompt stays small (typically 5-15 names
   for a normal process) instead of carrying all 460.

2. Post-process drift check — use the filtered list.
   The drift detector that warns "Heads up: the AI may have dropped
   detail" now uses the SAME filtered-to-source list as its
   SPECIFIC_NOUNS array. So if the source contains "SAMS" and the
   rewrite drops it, the warning toast names it explicitly.

How matching works:
  - Case-insensitive
  - Underscore = space (so "Adobe_Acrobat_DC" matches "Adobe Acrobat DC")
  - De-duped on the normalised form (we don't warn twice for variants)
