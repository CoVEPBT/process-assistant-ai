Parser robustness: markdown headings + capital-letter activities

Tracker #243 (4 Jun 2026): Testers pasted content using formats
PAT didn't recognise (markdown from Confluence/Notion, capital-letter
outlines). Either the parse came back empty or the AI-structuring
modal popped on content that already had perfectly good structure.

1. New parser handlers in parseStructuredText
=============================================
- markdownHeadingRe: /^(#{1,4})\s+(.+?)\s*#*\s*$/
  Single '# Title' becomes the process title (if not already set).
  '##' / '###' / '####' become activities, auto-numbered N.0 from
  the running activity count. Catches content pasted from
  Confluence, Notion, GitHub issues, README-style sources.
- capitalLetterActRe: /^([A-Z])[\.\)]\s+(.+)$/
  'A. Title', 'B) Title' style. Letter converted to ordinal so
  'B.' becomes activity 2.0. Common in legal / contractual / outline
  content.

Both handlers run after activityRe and stepActivityRe in the main
parse loop, so existing numeric formats are unchanged. Tasks under
the new activity styles still use the existing taskRe / numericTaskRe
/ bulletTaskRe matchers.

Verified with 4 unit cases: markdown h1+h2 with tasks, markdown
h2-only (no title), capital letters only, hybrid h1+capital letters.
All produce correct title + activity numbering + task assignment.

2. looksLikeRoughNotes stays consistent
=======================================
Without this, parseStructuredText would parse the markdown / capital
letter content fine but looksLikeRoughNotes would still trip first
and offer "want me to structure this?" on already-structured input.

Added markdown (#{1,4}\s+\S) and capital-letter ([A-Z][.)]\s+\S)
branches to the activityPattern regex. Modal now only triggers on
genuinely prose-only input, matching the new parser surface.
