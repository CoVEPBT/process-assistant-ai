PDF parser feedback item #1 follow-up: NOTE-truncation guardrails

Item #1 from the Training Plans (Apprentices & Trainees) test:
the AI shortened the AHPRA monthly-submission task and moved the
trigger condition into a NOTE - but truncated the NOTE body to
just "Each month after the student reporting is complete",
dropping the "...AND a formal acknowledgement of the submission
is received from AHPRA" half. The dropped clause is the actual
trigger condition, so losing it changes when the task runs.

Three changes to keep the AI from doing that again, and to stop
the drift-detection toast from false-positiving when the AI
DOES promote content into a NOTE correctly.

1. server/index.js SYSTEM_PROMPT
   Extended the "Promote dropped context to a NOTE" rule with an
   explicit "The NOTE body MUST keep the ENTIRE dropped detail -
   verbatim where possible" requirement, and a worked example
   built around the AHPRA "and a formal acknowledgement..."
   conjunction-chain pattern. Now matches the longer guidance
   already in the frontend prompt.

2. public/index.html PROMOTE DROPPED CONTEXT block
   (Already in place from earlier work - this commit keeps it
   in sync with the server prompt.) Adds the same AHPRA worked
   example and the "NOTE is not a place to keep dropping content"
   rule into the frontend auto-apply prompt.

3. public/index.html drift-detection haystack
   The post-auto-apply "Heads up: the AI may have dropped some
   detail" toast was scanning only the new TASK text for
   parentheticals / step refs / specific nouns. That false-
   positived whenever the AI did the right thing and moved
   detail into a NOTE under the same activity.

   Now builds a per-activity combined haystack:
     new task texts + new NOTE titles + new NOTE bodies
   and checks each dropped-detail candidate against that
   combined string. So "I moved 'Staff Name column' from
   the task into a NOTE" no longer trips the warning, but
   "I deleted 'Staff Name column' entirely" still does.
