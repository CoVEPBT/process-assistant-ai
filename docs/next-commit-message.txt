Fix banner filter dropping NOTE bodies + bare task letters

The banner filter I added in 291b3bd (PDF parser items #2 + #4
+ banner filter) was counting how many TIMES a candidate line
appeared across the whole document, and flagging anything that
appeared 2+ times as a "page banner" to drop.

That over-matched badly on two-column PDFs:
  - PDF.js sometimes emits single-letter "lines" (a, A, S, T,
    V, etc.) from letter-spaced text. With bare letters from
    task markers added in, "a" appeared many times. Counted as
    a banner. Dropped. So bare task letters never made it to
    the parser, and the bare-letter-merge step had nothing to
    merge.
  - Wrapped NOTE-body fragments and Y-bucket collisions
    (e.g. "Enrolment Domingo" - a left-column word colliding
    in Y-bucket with a right-column word) also got flagged.
    Some real NOTE-body lines got dropped, so the body never
    reached the NOTE during parseProcedureBlock continuation.

User-visible symptom: when uploading a multi-page two-column
Nintex PDF, NOTEs showed only their titles - bodies came back
empty even though they were clearly in the source.

Fix: instead of counting occurrences, track which PAGES each
candidate line appears on. Only treat a line as a banner if it
appears on 2+ DISTINCT pages. A column-collision artefact or a
wrapped sentence fragment on a single page no longer gets
filtered. The original target (title repeating between pages)
still gets caught - it appears on every page by design.

Verified against three PDFs:
  - 4-page Training Plans (two-column, has NOTE bodies):
      6 banners caught (real ones - title wrap + "None Noted"
      placeholder). NO bare letters or task fragments dropped.
      All NOTE bodies now parse correctly.
  - 3-page Training Plans (older single-column, title-only
    NOTEs): 2 banners caught - the original target
    "Student Training Plans (Apprentices & Trainees)" still
    drops as intended.
  - AHPRA PDF: 2 banners caught (title wrap parts), NOTE
    bodies still parse cleanly.
